# ナラティブパターン集

プレゼンテーションの構成を決めるための8つの定番パターン。ヒアリング結果（対象者・目的・トーン）に基づいて最適なパターンを選択する。

## パターン選択ガイド

| 対象者 + 目的 | 推奨パターン |
|-------------|------------|
| 経営層 + 意思決定を求める | 説得・提案 |
| 経営層 + 状況報告 | ステータスレポート |
| マネージャー + 改善提案 | 課題→解決 |
| マネージャー + 進捗報告 | ステータスレポート |
| エンジニア + 知識共有 | What→Why→How |
| エンジニア + ツール選定 | 比較・評価 |
| 顧客 + サービス紹介 | オンボーディング |
| 全般 + ビジョン・戦略 | 現状→未来 |
| 全般 + 複雑な問題の整理 | SCR |

迷った場合は「課題→解決」が汎用性が高い。

---

## 1. 課題→解決

最も汎用的なパターン。現状の課題を提示し、解決策を提案する流れ。

**向いている場面:** 改善提案、ツール導入提案、プロセス変更提案

**スケルトン（10枚標準）:**

| # | スライド種別 | キーメッセージ |
|---|-----------|-------------|
| 1 | タイトル | [提案のタイトル] |
| 2 | 背景・現状 | 今こうなっている（事実ベース） |
| 3 | 課題提起 | ここに問題がある（データ・事例で裏付け） |
| 4 | 影響・インパクト | 放置するとどうなるか（コスト・リスク） |
| 5 | 解決策の概要 | こうすれば解決できる（1スライドで全体像） |
| 6 | 解決策の詳細1 | 具体的な施策・機能・手順 |
| 7 | 解決策の詳細2 | 追加の施策・導入ステップ |
| 8 | 期待される効果 | 定量的な改善見込み（数値で示す） |
| 9 | スケジュール・次のステップ | いつ・誰が・何をするか |
| 10 | まとめ | 要点3つに絞って再提示 |

**5-8枚に短縮する場合:** Slide 6-7を1枚に統合、Slide 4を削除
**15-20枚に拡張する場合:** 課題を複数スライドに分割、解決策の詳細を増やす、事例・ベンチマークを追加

---

## 2. 現状→未来

現在地と目指す姿のギャップを示し、そこに至る道筋を描く。ビジョン共有に最適。

**向いている場面:** 中長期戦略、ロードマップ共有、組織変革の提案

**スケルトン（12枚標準）:**

| # | スライド種別 | キーメッセージ |
|---|-----------|-------------|
| 1 | タイトル | [ビジョン・テーマ] |
| 2 | 現状の全体像 | 今のポジション・状況を客観的に描写 |
| 3 | 現状の強み | 活かせるアセット・実績 |
| 4 | 現状の課題 | 解決すべきギャップ |
| 5 | 目指す未来像 | こうなりたい（具体的なゴール像） |
| 6 | ギャップの分析 | 現状と未来の間に何があるか |
| 7 | アプローチ1 | 第一の施策・フェーズ |
| 8 | アプローチ2 | 第二の施策・フェーズ |
| 9 | アプローチ3 | 第三の施策・フェーズ |
| 10 | ロードマップ | 時系列で全体の流れ |
| 11 | 期待される成果 | 未来像が実現した時の姿 |
| 12 | まとめ・次のステップ | 最初の一歩は何か |

---

## 3. SCR（状況→複雑化→解決）

マッキンゼー式のストーリーライン。複雑な問題を整理して伝えるのに向いている。

**向いている場面:** 経営層への報告、コンサルティング提案、複雑な意思決定

**スケルトン（8枚標準）:**

| # | スライド種別 | キーメッセージ |
|---|-----------|-------------|
| 1 | タイトル | [テーマ] |
| 2 | 状況（Situation） | 関係者が合意している前提・事実 |
| 3 | 複雑化（Complication） | 何が変わったか・何が問題か |
| 4 | 問い（Question） | だからどうすべきか？（暗黙でも明示でも可） |
| 5 | 解決策（Resolution） | 結論を先に述べる |
| 6 | 根拠1 | 解決策を支える理由・データ |
| 7 | 根拠2 | 追加の根拠・事例 |
| 8 | 次のステップ | 具体的なアクション |

**ポイント:** 結論を先に述べるのがSCRの本質。根拠は結論の後に配置する。

---

## 4. What→Why→How

知識共有・教育に最適。「何か」を定義し、「なぜ重要か」を説明し、「どう使うか」を示す。

**向いている場面:** 技術共有LT、チーム勉強会、ツール紹介、新技術の導入説明

**スケルトン（8枚標準）:**

| # | スライド種別 | キーメッセージ |
|---|-----------|-------------|
| 1 | タイトル | [技術・ツール・概念名] |
| 2 | What: 定義・概要 | これは何か（1文で説明できるレベル） |
| 3 | What: 仕組み・構成 | どう動くか・何で構成されるか |
| 4 | Why: 背景・動機 | なぜこれが必要か・何を解決するか |
| 5 | Why: メリット | 使うとどう良くなるか（定量データがあれば） |
| 6 | How: 基本的な使い方 | 最小限の手順・コード例 |
| 7 | How: 実践Tips | チームでの活用方法・注意点 |
| 8 | まとめ・参考リンク | 要点と学習リソース |

**5枚に短縮する場合:** What/Why/Howを各1枚に圧縮
**15-20枚に拡張する場合:** デモスライド、比較表、FAQ、ハンズオン手順を追加

---

## 5. ステータスレポート

定期的な進捗報告。事実ベースで簡潔に、次のアクションを明確にする。

**向いている場面:** 週次・月次報告、プロジェクト進捗、四半期レビュー

**スケルトン（8枚標準）:**

| # | スライド種別 | キーメッセージ |
|---|-----------|-------------|
| 1 | タイトル | [期間] [プロジェクト名] 進捗報告 |
| 2 | エグゼクティブサマリー | 3行で全体像（順調/注意/要対応） |
| 3 | KPI・指標 | 数値で現状を示す |
| 4 | 今期の実績 | 完了したこと・達成したこと |
| 5 | 進行中の取り組み | 現在進めていること |
| 6 | 課題・リスク | ブロッカーや懸念事項 |
| 7 | 次期のアクション | 次にやること・マイルストーン |
| 8 | まとめ | 判断・支援が必要な事項があれば明記 |

---

## 6. 説得・提案

意思決定者に承認・予算・リソースを求める。論理的な根拠と感情的な訴求のバランスが重要。

**向いている場面:** 予算申請、新規プロジェクト提案、方針転換の提案

**スケルトン（12枚標準）:**

| # | スライド種別 | キーメッセージ |
|---|-----------|-------------|
| 1 | タイトル | [提案名] |
| 2 | 現在の痛み | 今困っていること（共感を得る） |
| 3 | 放置のコスト | 何もしないとどうなるか（損失・リスク） |
| 4 | 提案の概要 | 何を提案するか（1文で） |
| 5 | 提案の詳細 | 具体的な内容・スコープ |
| 6 | 根拠・データ | 提案を支える定量的根拠 |
| 7 | 事例・実績 | 他社事例や過去の成功例 |
| 8 | 代替案との比較 | なぜこの案がベストか |
| 9 | 導入計画 | ステップ・スケジュール |
| 10 | 投資対効果 | コストとリターン |
| 11 | リスクと対策 | 想定リスクと軽減策 |
| 12 | 依頼事項 | 何を承認してほしいか（明確に） |

---

## 7. 比較・評価

複数の選択肢を公平に評価し、推奨を示す。ツール選定や技術選定に最適。

**向いている場面:** ツール・ベンダー選定、アーキテクチャ比較、方式検討

**スケルトン（12枚標準）:**

| # | スライド種別 | キーメッセージ |
|---|-----------|-------------|
| 1 | タイトル | [比較テーマ] |
| 2 | 背景・目的 | なぜ比較が必要か |
| 3 | 評価軸 | 何を基準に比較するか（3-5軸） |
| 4 | 選択肢A の概要 | 特徴・強み・弱み |
| 5 | 選択肢B の概要 | 特徴・強み・弱み |
| 6 | 選択肢C の概要 | （必要な場合） |
| 7 | 比較表 | 評価軸 x 選択肢のマトリクス |
| 8 | コスト比較 | TCO・ライセンス・運用コスト |
| 9 | 技術的フィット | 既存環境との相性 |
| 10 | リスク評価 | 各選択肢のリスク |
| 11 | 推奨 | どれをなぜ推奨するか |
| 12 | 次のステップ | PoC計画・導入スケジュール |

**ポイント:** 評価軸を先に定義してから各選択肢を評価する。結論ありきに見えないよう、プロセスの透明性を示す。

---

## 8. オンボーディング

新しいメンバーや顧客に対して、組織・製品・プロジェクトを紹介する。

**向いている場面:** 新メンバー向け説明、顧客への製品紹介、チーム紹介

**スケルトン（10枚標準）:**

| # | スライド種別 | キーメッセージ |
|---|-----------|-------------|
| 1 | タイトル | [紹介対象名] |
| 2 | 全体像・ミッション | 何のために存在するか |
| 3 | 歴史・経緯 | どうしてこうなったか（簡潔に） |
| 4 | 主要機能・サービス1 | コアバリュー |
| 5 | 主要機能・サービス2 | 差別化ポイント |
| 6 | 主要機能・サービス3 | 補完的な価値 |
| 7 | ユースケース・事例 | 実際にどう使われているか |
| 8 | よくある質問 | 初見で疑問に思うこと |
| 9 | 始め方・次のステップ | 最初に何をすればいいか |
| 10 | 連絡先・リソース | 困ったときの問い合わせ先 |
